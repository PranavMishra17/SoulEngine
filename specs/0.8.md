# Spec 0.8 — Serialize local usage append

**Backlog item:** 0.8 (Tier 0 — Critical Blocker)  
**Error ledger:** ERR-008  
**Problem:** `appendProjectUsage` in `src/storage/local/usage.ts` performs an unlocked read-modify-write, causing concurrent calls for the same project to lose updates.

## Acceptance Criteria

1. Concurrent calls to `appendProjectUsage` for the same project do not lose updates: N concurrent appends are ALL reflected in the final persisted totals.
2. The public interface and read semantics are unchanged; `getProjectUsage` still returns correct aggregates.
3. A regression test at `tests/regression/err-008-usage-append-race.test.ts` fires many concurrent `appendProjectUsage` calls and asserts the final total equals the sum.
4. The test must FAIL against the current code (demonstrating the bug) and PASS after the fix.

## Root Cause

`appendProjectUsage` (lines 41-66):
1. Reads current totals via `await getProjectUsage(projectId)`.
2. Merges with new session usage.
3. Writes the updated totals via `await fs.writeFile(...)`.

When two concurrent calls execute for the same project:
- Both read the same old totals (e.g., `{total_conversations: 5, text_input_tokens: 1000}`).
- Call A calculates `{total_conversations: 6, text_input_tokens: 1050}`.
- Call B calculates `{total_conversations: 6, text_input_tokens: 1100}`.
- The second write clobbers the first — one increment is lost.

## Chosen Approach

Serialize per-project writes using an in-process async lock. Specifically:

- Maintain a `Map<projectId, Promise<void>>` that tracks the tail promise for each project.
- Before starting a read-modify-write, each call chains onto the tail promise for that project.
- This ensures writes for a given project execute sequentially, while different projects can still run concurrently.

This fixes single-process concurrency (the realistic deployment case). Cross-process safety requires the cloud backend (which already implements atomic RPC) — a code comment notes this.

## Test Plan

**Regression test** (`tests/regression/err-008-usage-append-race.test.ts`):

1. **Setup:** Create a temp directory for test data; override `DATA_DIR` for the test.
2. **Action:** Fire N=20 concurrent `appendProjectUsage` calls for the same project, each adding distinct token amounts (e.g., call i adds `i` tokens).
3. **Assert:** The final `getProjectUsage` total equals the sum of all increments (e.g., sum(1..20) = 210 text_input_tokens).
4. **Fail-before evidence:** The test must fail against the current code (final total < expected due to lost updates).
5. **Pass-after evidence:** The test passes after the lock is added.
6. **Cleanup:** Remove the temp directory after the test.

## Security / Correctness Notes

- This is an observability/accounting bug, not a security hole, but it undermines usage-based billing and quota enforcement.
- The fix maintains backward compatibility: the public API is unchanged.
- Error handling: the existing `try/catch` and non-fatal logging remain unchanged; the lock does not interfere with error propagation.

# ERR-006: Make encrypted secrets rotatable and portable across backends

## Problem
The secret encryption code is duplicated in `src/storage/local/secrets.ts` (whole-object JSON format) and `src/storage/supabase/secrets.ts` (per-key colon-delimited strings). The two implementations share crypto constants but duplicate the encrypt/decrypt logic. Neither format includes a key-version tag, so rotating the `ENCRYPTION_KEY` permanently bricks all stored secrets when `decipher.final()` fails on the old ciphertext.

## Acceptance Criteria
1. A single shared crypto module (`src/storage/crypto/secrets.ts`) provides `encrypt`/`decrypt` (or `seal`/`open`) used by BOTH backends. The serialized envelope includes a `keyVersion` field (and algorithm tag) so future rotations are detectable.
2. A documented rotation routine exists (a function, even if not yet wired to an endpoint): given old and new keys, decrypt-with-old then re-encrypt-with-new.
3. Round-trip correctness: encrypt then decrypt returns the original; preserve PBKDF2 + random salt + authTag (no IV reuse). Maintain backward-compatibility with existing stored blobs OR provide a clear, tested migration path.
4. `npm run build` clean; `npm test` green.

## Chosen Approach
Create `src/storage/crypto/secrets.ts` that:
- Exports `encryptSecret(plaintext, encryptionKey, keyVersion?)` and `decryptSecret(envelope, encryptionKey)`.
- Uses the existing AES-256-GCM + PBKDF2 params (100k iterations, 32-byte salt).
- Serializes to a versioned JSON envelope: `{algorithm, keyVersion, salt, iv, authTag, ciphertext}`.
- Provides `rotateSecret(envelope, oldKey, newKey)` that decrypts with old, re-encrypts with new, incrementing `keyVersion`.
- Gracefully handles decryption failures (wrong key, tampered tag) by throwing a typed `DecryptionError` that doesn't leak material.

Refactor both backends to call the shared module:
- `local/secrets.ts`: continue using whole-object encryption (ApiKeys → JSON → encrypt), but call the shared encrypt/decrypt.
- `supabase/secrets.ts`: continue encrypting per-key (each string value → encrypt), but call the shared encrypt/decrypt for each field.

Backward-compatibility: the shared module will detect old formats (local: check for `version: 1` + old shape; supabase: colon-delimited string) and migrate on first read, or provide a migration helper if needed.

## Files to Touch
- `src/storage/crypto/secrets.ts` (new shared module)
- `src/storage/local/secrets.ts` (refactor to use shared module)
- `src/storage/supabase/secrets.ts` (refactor to use shared module)
- `tests/regression/err-006-secrets-roundtrip.test.ts` (TDD test, write FIRST)

## Test Plan
1. **Round-trip correctness** (tests/regression/err-006-secrets-roundtrip.test.ts):
   - Encrypt a plaintext with key A, version 1 → decrypt with key A returns original.
   - Encrypt an object (ApiKeys-like) → decrypt returns the original object.
   - Encrypt with key A v1, rotate to key B v2 → decrypt with B returns original, decrypt with A fails gracefully.
   - Decrypt with wrong key throws clear error, no crash, no leaked material in message.
   - Tampering with authTag → decrypt fails with integrity error.

2. **Backward-compatibility** (same test file or separate migration test):
   - Local old format (`{iv, salt, authTag, data, version: 1}`) → decrypt via shared module returns original.
   - Supabase old format (`salt:iv:authTag:data`) → decrypt via shared module returns original.

3. **Integration** (existing tests should remain green):
   - `npm test` runs existing storage tests; they should pass with the refactored backends.
